# 0. 先建立 SDR 的核心定位

**SDR 是 I3C 的基础控制平面。**

即使系统以后要跑 HDR-DDR、HDR-BT，也必须先从 SDR 进入。总线初始化、动态地址分配、CCC、IBI、Hot-Join、Controller Role Request 等核心管理动作，都是围绕 SDR 建立的。

可以把 SDR 理解成：

```text
I3C SDR = I2C-like 外形 + I3C 专用语义

外形像 I2C：
START / STOP / 7-bit Address / RnW / ACK

语义不像普通 I2C：
动态地址
广播地址 7'h7E
CCC
IBI
Hot-Join
第 9 bit 不是传统 ACK
Push-Pull 高速传输
START 后地址仲裁
```

---

# 1. SDR 总线配置：谁在总线上，谁能控制总线

I3C 总线最基本的结构是：

```text
I3C Bus
 ├── Active Controller      当前真正控制总线的设备
 ├── I3C Target             普通 I3C 目标设备
 ├── Secondary Controller   平时是 Target，有 Controller 能力
 └── Legacy I2C Target      可选共存的传统 I2C 设备
```

## 1.1 同一时刻只能有一个 Active Controller

这点非常重要：

```text
Primary Controller ≠ 永远控制总线
Active Controller  = 当前控制总线
Secondary Controller = 未来可能接管总线的 Target
```

初始化时，Primary Controller 是第一个 Active Controller。后续如果有 Secondary Controller，它可以通过 Controller Role Request 请求接管总线。但任意时刻，总线上只能有一个 Active Controller。

## 1.2 I3C Target 至少要支持 SDR

一个 I3C Target 的基础义务是：

```text
识别 7'h7E
识别自己的 Dynamic Address
响应 Required CCC
支持至少一种动态地址分配方式
支持 SDR Mode
```

是否支持 HDR 是额外能力。SDR-Only Target 也仍然是合法 I3C Target。

## 1.3 总线配置还要区分 Pure Bus 和 Mixed Bus

I3C 总线可以有三类配置：

|总线类型|含义|对协议影响|
|---|---|---|
|Pure Bus|只有 I3C 设备|可用更完整的 I3C 高速能力|
|Mixed Fast Bus|I3C + 带 50ns Spike Filter 的 Legacy I2C Target|I3C 可用短 SCL high 让 I2C 设备“看不见”I3C 流量|
|Mixed Slow/Limited Bus|I3C + 不具备合适滤波能力的 Legacy I2C Target|速度和模式受限更多|

对驱动/硬件 bring-up 来说，这意味着：**你不能只看 I3C Controller 和 Target，还要知道总线上有没有 Legacy I2C 设备，以及它们的 LVR 能力。**

## 1.4 BCR / DCR / LVR 是总线配置的基础信息

I3C 设备通过特征寄存器描述能力：

|寄存器|作用|
|---|---|
|BCR|Bus Characteristics Register，描述设备角色、IBI 能力、HDR/高级能力、是否 Controller-capable 等|
|DCR|Device Characteristics Register，描述设备类型，例如传感器、通用设备等|
|LVR|Legacy Virtual Register，描述 Legacy I2C 设备能力，由系统/驱动虚拟维护|

你做驱动时可以这样理解：

```text
BCR 决定“这个设备能干什么”
DCR 决定“这个设备是什么类型”
LVR 决定“这个 I2C 老设备会限制总线到什么程度”
```

# 2. SDR 总线数据传输：Frame / Message / Header / Data

SDR 的基本结构仍然是：

```text
START → HEADER → DATA → STOP
```

但要再细分两个概念：

```text
Frame   = 从 START 到 STOP 的完整总线占用
Message = 从 START 或 Repeated START 到下一个 Repeated START 或 STOP
```

一个 Frame 里面可以有多个 Message，因为中间可以有 Repeated START。

---

## 2.1 Header 格式

SDR Header 和 I2C 外形一致：

```text
7-bit Address + RnW + ACK/NACK
```

例如：

```text
S → Address[6:0] → RnW → ACK/NACK → DATA...
```

但是 I3C 有一个关键差异：

```text
START 后的 Header 是可仲裁的
Repeated START 后的 Header 不仲裁，通常 Push-Pull
```

也就是：

|位置|是否仲裁|驱动方式|
|---|---|---|
|START 后 Header|可仲裁|Open-Drain 起始，可能优化成 Push-Pull|
|Repeated START 后 Header|不仲裁|Address + RnW 用 Push-Pull，ACK/NACK 仍 Open-Drain|

为什么 START 后要仲裁？因为 Target 也可能趁这个机会发起：

```text
IBI
Hot-Join
Controller Role Request
```

所以 START 后不仅仅是 Controller 发地址，也可能是 Target 在抢总线表达请求。

---

## 2.2 SDR 有四种典型传输

### 传输 1：Broadcast CCC

```text
S
7'h7E + W + ACK
CCC
Optional Data
P / Sr
```

这是给所有 I3C Target 的命令。`7'h7E` 是 I3C Broadcast Address，所有 I3C Target 都要识别。

### 传输 2：Direct CCC

Direct CCC 也先从 `7'h7E` 开始，因为 Controller 需要先告诉所有 Target：“接下来是一个 Direct CCC”。

典型格式：

```text
S
7'h7E + W + ACK
Direct CCC
Sr
Target Dynamic Address + R/W + ACK
Data / Response
P
```

Direct CCC 的重点是：

```text
命令码先广播给所有 Target
真正响应的是后面被 Dynamic Address 选中的 Target
```

### 传输 3：Private Write

普通写 Target：

```text
S / Sr
Target Dynamic Address + W + ACK
Data Byte + T
Data Byte + T
...
P / Sr
```

不过规范通常建议 Controller 从 `7'h7E` 开始 I3C Frame，或者在 Repeated START 后发 Target Dynamic Address，这样可以避免和 IBI/CRR 等 START 后仲裁请求混淆。

### 传输 4：Private Read

普通读 Target：

```text
S / Sr
Target Dynamic Address + R + ACK
Target Data Byte + T
Target Data Byte + T
...
P / Sr
```

Private Read 的结束不是传统 I2C 的 Controller ACK/NACK 模型，而是 I3C 的 T-bit 模型。

# 3. SDR Data Word：第 9 bit 是理解 I3C 的关键

I3C SDR 和 I2C 都是 9 个 SCL 周期一组数据，但第 9 bit 的语义完全不同。

## 3.1 Controller 写 Target：第 9 bit 是奇校验 T-bit

I2C 写数据时：

```text
8-bit data + Target ACK
```

I3C SDR 写数据时：

```text
8-bit data + Parity T-bit
```

Target 不在第 9 bit 拉 ACK。第 9 bit 是 Controller 发送的奇校验位。

```verilog
// I3C SDR write T-bit, odd parity
t_bit = ~(^data[7:0]);
```

也就是：

|Data 中 1 的个数|T-bit|总 1 个数|
|---|---|---|
|偶数|1|奇数|
|奇数|0|奇数|

所以你看波形时，不要再把 SDR Write 的第 9 bit 当 ACK 看，否则会误判。

---

## 3.2 Target 返回读数据：第 9 bit 是 End/Continue 语义

I3C SDR Read 时：

```text
8-bit data + T-bit
```

这个 T-bit 由 Target 驱动，用来表达是否继续。

简化理解：

|Target T-bit|含义|
|---|---|
|1|继续读，后面还有数据|
|0|Target 结束本次读|

Controller 也可以在合适时机通过 Repeated START 中止读。

所以 I3C Read 的长度可以由 Target 主动控制，这比 I2C 灵活很多。I2C 传统模型里，通常是 Controller 决定读多少字节；I3C 允许 Target 通过 T-bit 表达结束。

# 4. Bus Conditions：Free / Available / Idle

SDR 不只是 START/STOP，还定义了三个“总线空闲程度”。

```text
STOP 之后
  ↓
Bus Free
  ↓
Bus Available
  ↓
Bus Idle
```

## 4.1 Bus Free Condition

Bus Free 是 STOP 之后，经过最小时间，Controller 可以认为总线可再次发起正常传输。

对 Pure Bus 和 Mixed Bus，时间要求不同。Mixed Bus 要考虑 Legacy I2C 的 tBUF。

## 4.2 Bus Available Condition

Bus Available 比 Bus Free 更“空”。Target 只有在 Bus Available 之后，才可以主动发起 START Request，例如：

```text
IBI
Controller Role Request
```

所以如果你在验证 IBI，Target 不应该在任意空隙乱拉 SDA，它要等到 Bus Available。

## 4.3 Bus Idle Condition

Bus Idle 是更长时间的 SDA/SCL 均为 High，用于 Hot-Join 判断。

标准 Hot-Join Target 必须先确认 Bus Idle，才可以认为总线已经稳定、没有正在进行 HDR 或其他传输，然后发起 Hot-Join Request。

规范图里典型值有：

```text
tAVAL = 1.0 us
tIDLE = 200 us
```

驱动视角下，Hot-Join 不只是“设备上电就拉线”，它必须等待一个足够可靠的总线空闲条件。

# 5. 动态地址分配 DAA：I3C 初始化的核心流程

I3C 的普通通信依赖 Dynamic Address。初始化时，Primary Controller 要给每个 I3C Target 分配唯一动态地址。

动态地址来源有三类方法：

|方法|作用|
|---|---|
|SETDASA|根据已知 Static Address，给某个 Target 设置 Dynamic Address|
|SETAASA|让所有有 Static Address 的 I3C Target 使用 Static Address 作为 Dynamic Address|
|ENTDAA|通用动态地址分配流程，基于 PID/BCR/DCR 仲裁|

真正通用、最关键的是 **ENTDAA**。

---

## 5.1 ENTDAA 总流程

简化波形：

```text
S
7'h7E + W + ACK
ENTDAA CCC
Sr
7'h7E + R + ACK
Target 输出 PID/BCR/DCR 参与仲裁
Controller 分配 Dynamic Address + parity
Target ACK
重复，直到没有 Target ACK
P
```

更具体地说：

### Step 1：Controller 发 ENTDAA

```text
S → 7'h7E + W → ENTDAA
```

这个 Broadcast CCC 表示进入 Dynamic Address Assignment。

### Step 2：Controller 发 `7'h7E + R`

```text
Sr → 7'h7E + R
```

这点容易忽略：`7'h7E + R` 在普通 SDR 里不是一般广播读，而是 DAA 模式专用入口。

还没有 Dynamic Address、且支持 ENTDAA 的 Target 会 ACK。

### Step 3：Target 用 PID + BCR + DCR 仲裁

所有参与 DAA 的 Target 同时用 Open-Drain 发送：

```text
48-bit Provisioned ID
+ BCR
+ DCR
```

谁的拼接值最低，谁赢得本轮仲裁。

这和 I2C 仲裁类似：

```text
0 胜过 1
低值胜出
```

### Step 4：Controller 给赢家分配 Dynamic Address

```text
Controller → 7-bit Dynamic Address + parity
Winner Target → ACK
```

然后 Controller 回到 Step 2，继续给下一个 Target 分配地址。

### Step 5：没有 Target ACK 后结束

```text
P
```

---

## 5.2 Dynamic Address 还承载 IBI 优先级

I3C 的地址不只是“编号”，还隐含优先级：

```text
Dynamic Address 越小，IBI / CRR 优先级越高
```

原因是 START 后地址仲裁时，Open-Drain 中 `0` 赢 `1`。低地址更容易在高位上先出现 0，所以优先级更高。

因此 Controller 分配 Dynamic Address 时，应当考虑中断优先级。例如：

```text
高优先级 sensor / Secondary Controller → 分配较小 Dynamic Address
低优先级普通 Target             → 分配较大 Dynamic Address
```

---

## 5.3 Dynamic Address 的禁用范围

这些地址不能作为普通 Target Dynamic Address：

```text
7'h00
7'h01
7'h02
7'h7E
7'h7F
```

其中：

|地址|含义|
|---|---|
|`7'h02`|Hot-Join Address|
|`7'h7E`|I3C Broadcast Address|
|`7'h7F`|Reserved|

此外，一些与 `7'h7E` 只差一位的地址也不能用，例如：

```text
7'h3E
7'h5E
7'h6E
7'h76
7'h7A
7'h7C
7'h7F
```

目的是避免 Target 把单 bit 错误的地址误判成 Broadcast Address。

# 6. Hot-Join：设备后加入总线

Hot-Join 解决的是：

```text
I3C Bus 已经初始化完成
某个 Target 后上电 / 后插入 / 后恢复
它如何加入总线并拿到 Dynamic Address？
```

Hot-Join 的关键点：

```text
Hot-Join 只能发生在总线已经配置之后
Hot-Join Request 使用 reserved address 7'h02
Hot-Join 之后仍然通过 ENTDAA 分配 Dynamic Address
```

## 6.1 Hot-Join Request 外形

Hot-Join Request 类似 IBI，但地址特殊：

```text
START
7'h02 + W
ACK/NACK by Active Controller
```

注意：

```text
Hot-Join Request 没有 Mandatory Data Byte
```

它只是告诉 Controller：

```text
我来了，请稍后给我分配 Dynamic Address
```

## 6.2 Controller 如何响应 Hot-Join

Controller 有两种响应：

|响应|含义|
|---|---|
|ACK|我看到了，后面会通过 ENTDAA 给你地址|
|NACK|暂时拒绝，你以后再试|

ACK 不代表马上 ENTDAA。Controller 可以先 ACK，然后过一段时间再发 ENTDAA。

## 6.3 Hot-Join 可以被 CCC 使能/禁止

通过 ENEC / DISEC CCC 控制：

|CCC|bit|含义|
|---|---|---|
|ENEC|ENHJ|Enable Hot-Join|
|DISEC|DISHJ|Disable Hot-Join|

这对系统很有用。例如 Controller 当前资源不足、不想处理新设备，就可以先 DISHJ。

## 6.4 Standard Hot-Join vs Passive Hot-Join

|类型|行为|
|---|---|
|Standard Hot-Join|等 Bus Idle，然后可以主动拉 SDA 发 START Request|
|Passive Hot-Join|先观察到一个 SDR Frame 以 STOP 结束，再等下一个 START，确认自己确实在 I3C Bus 上|

Passive Hot-Join 对避免误判 Legacy I2C Bus 有帮助。

# 7. IBI：In-Band Interrupt，带内中断

IBI 是 I3C 相比 I2C 很关键的改进。传统 I2C 设备常常需要额外 GPIO interrupt pin；I3C 允许 Target 直接在 SDA/SCL 两线内发中断请求。

## 7.1 IBI 基本流程

Target 要发 IBI 时，在 START 后的可仲裁 Header 中发送：

```text
Target Dynamic Address + RnW = 1
```

也就是：

```text
S
Target_DA + R
Controller ACK/NACK
Optional Mandatory Data Byte
Optional additional IBI payload
P / Sr
```

如果当前没有 START，但 Bus 已经 Available，Target 可以拉低 SDA 请求 START，Controller 随后拉低 SCL 并提供时钟。

## 7.2 多个 Target 同时 IBI 怎么办？

靠地址仲裁。

```text
低 Dynamic Address 胜出
高 Dynamic Address 输掉后等待下一次 Bus Available 再尝试
```

所以 IBI 是天然优先级中断。

## 7.3 Controller 对 IBI 的三种处理

|Controller 行为|含义|
|---|---|
|ACK|接受 IBI|
|NACK|暂时拒绝，Target 可后续重试|
|NACK + DISEC.DISINT|拒绝并关闭该 Target 的中断请求能力|

## 7.4 Mandatory Data Byte

如果 Target 的 BCR[2] = 1，表示 IBI 后面带 payload。Controller ACK IBI 后，必须接收第一个 Mandatory Data Byte。

这个字节用于告诉 Controller 中断类型，例如：

```text
MDB[7:5] = Interrupt Group Identifier
MDB[4:0] = Specific Interrupt Identifier
```

之后 Target 还可以继续发送额外 IBI 数据，长度受 MRL/IBI payload 相关配置约束。

# 8. CCC：Common Command Code，I3C 的标准管理命令集

CCC 是 I3C 的标准控制命令。

一句话：

```text
CCC = Controller 管理 I3C Bus 和 Target 的标准命令
```

SDR 模式下，CCC 永远从 `7'h7E + W` 开始：

```text
S / Sr
7'h7E + W + ACK
CCC Code
...
```

## 8.1 Broadcast CCC vs Direct CCC

|类型|Command Code 范围|作用|
|---|---|---|
|Broadcast CCC|`0x00 ~ 0x7F`|发给所有 I3C Target|
|Direct CCC|`0x80 ~ 0xFE`|先广播命令码，再定向访问某个 Target|
|Reserved|`0xFF`|保留|

Target 看到 CCC Code 的 bit[7] 就能判断：

```text
bit[7] = 0 → Broadcast CCC
bit[7] = 1 → Direct CCC
```

---

## 8.2 Broadcast CCC 格式

```text
S / Sr
7'h7E + W + ACK
Broadcast CCC + T
Optional Defining Byte + T
Optional Data + T
P / Sr
```

Broadcast CCC 所有 Target 都要检查。即使某个 Target 不支持某个 Optional CCC，也要能正确识别边界并忽略。

---

## 8.3 Direct CCC 格式

Direct CCC 的关键是：**先广播命令码，再 Repeated START 到目标地址。**

```text
S / Sr
7'h7E + W + ACK
Direct CCC + T
Optional Defining Byte + T
Sr
Target Dynamic Address + R/W + ACK
Data / Response
P
```

一个 Direct CCC 也可以连续访问多个 Target：

```text
7'h7E + W
Direct CCC
Sr TargetA ...
Sr TargetB ...
Sr TargetC ...
P
```

## 8.4 常用 CCC 分类

对驱动开发最重要的 CCC 可以按功能记：

|功能|CCC|
|---|---|
|事件控制|ENEC / DISEC|
|动态地址|RSTDAA / ENTDAA / SETDASA / SETAASA / SETNEWDA|
|能力查询|GETPID / GETBCR / GETDCR / GETSTATUS / GETCAPS|
|传输能力|SETMWL / GETMWL / SETMRL / GETMRL / GETMXDS|
|HDR 入口|ENTHDR0 / ENTHDR3 等|
|Secondary Controller|DEFTGTS / GETACCCR|
|Target Reset|RSTACT|
|Group Address|SETGRPA / RSTGRPA / DEFGRPA|

你在写 I3C Controller 驱动时，初始化阶段通常离不开：

```text
RSTDAA
SETDASA / SETAASA / ENTDAA
GETBCR
GETDCR
GETPID
GETSTATUS
ENEC / DISEC
SETMRL / SETMWL
```

# 9. 额外值得关注：Clock Stalling

SDR 里 Controller 可以在特定场景拉住 SCL Low，叫 **Clock Stalling**。

它不是 I2C clock stretching。I3C 中 SCL 主要由 Controller 控制，Target 通常不能随意 stretch clock。

Controller 可以 Stall 的典型位置：

```text
ACK/NACK phase
Write data parity bit
Read data T-bit
ENTDAA 分配地址的第一 bit 前
```

用途：

```text
Controller FIFO underrun
Controller FIFO overflow
等待内部状态机
根据 BCR/DCR 计算 Dynamic Address
```

对 HAPS/EMU 或硅后调试来说，如果你看到 SCL Low 被 Controller 拉长，不一定是异常；要看它是否发生在规范允许的 phase。

# 10. 从驱动视角看 SDR 初始化主流程

一个典型 I3C Controller bring-up 可以抽象成：

```text
1. 配置 Controller PHY / timing / pull-up / high-keeper

2. 进入 SDR 初始状态

3. 发送第一个 7'h7E + W
   让 I3C Target 识别自己在 I3C Bus 上
   某些 Target 会因此关闭 I2C spike filter

4. RSTDAA
   清除旧 Dynamic Address 状态

5. 根据已知静态地址做 SETDASA / SETAASA
   给部分已知设备快速分配地址

6. ENTDAA
   给剩余 Target 分配 Dynamic Address

7. 读取能力
   GETPID / GETBCR / GETDCR / GETSTATUS / GETCAPS

8. 配置传输限制
   SETMRL / SETMWL / GETMXDS

9. 配置事件
   ENEC：打开 IBI / Hot-Join / CRR
   或 DISEC：按策略关闭某些事件

10. 正常 Private Read/Write 或 CCC 管理
```

```text
│
├── Bus Conditions

│

    └── GETPID / GETBCR / GETDCR / GETSTATUS / GETCAPS
```

一句话总结：**SDR 是 I3C 的“管理态 + 基础传输态”。先通过 SDR 完成总线发现、地址分配、能力枚举和事件配置，再通过 SDR Private Transfer 做普通读写，必要时通过 CCC 进入 HDR 或处理更复杂的总线管理。**

# 答疑
## I3C 和 I2C 最大的三个区别是什么？
按驱动和波形理解，**I3C 和 I2C 最大的三个区别**可以抓这三条：

### 1. 地址机制不同：I2C 靠静态地址，I3C 核心靠动态地址

I2C 设备通常用固定的 7-bit Static Address。  
I3C 虽然兼容部分 Legacy I2C Target，但 I3C Target 正常工作时会由 Controller 分配 **Dynamic Address**。

I3C 初始化时常见流程是：

```
Controller → 7'h7E 广播 → ENTDAA / SETDASA / SETAASA → 给 Target 分配 Dynamic Address
```

所以 I3C 驱动里会有一个重要数据结构：**Dynamic Address Table**。后续普通读写一般访问的是 Target 的 Dynamic Address，而不是传统 I2C 静态地址。规范也明确说 Primary Controller 在每次上电启动时要给总线上每个 active role 分配唯一动态地址。

---

### 2. 传输电气/时序不同：I2C 主要 Open-Drain，I3C 大量使用 Push-Pull

I2C 的典型传输是 Open-Drain + 上拉电阻，速度受 RC 上升沿限制明显。

I3C SDR 外形像 I2C，但差别很大：

```
START / STOP / 7-bit Address / RnW / ACK 这些像 I2C但数据阶段大量用 Push-Pull，高速得多
```

还有一个非常容易踩坑的点：**I3C SDR 的第 9 bit 不再总是 I2C 那种 ACK/NACK。**

Controller 写 Target 时：

```
8-bit Data + T-bit
```

这里 T-bit 是奇校验 parity，不是 Target ACK。

Target 返回读数据时：

```
8-bit Data + T-bit
```

这里 T-bit 表示继续/结束/读传输控制。  
所以抓 I3C 波形时，不能把所有第 9 bit 都按 I2C ACK/NACK 解释。规范也说明 I3C SDR Data Word 和 I2C 都是 9 bit，但第 9 bit 语义不同。

---

### 3. 总线管理能力不同：I3C 有 IBI、Hot-Join、CCC、多 Controller 角色切换

I2C 的从设备通常不能在总线内主动“中断”主机，常常需要额外 GPIO interrupt pin。I3C 把很多管理能力放进两线协议内：

```
IBI       = In-Band Interrupt，Target 带内中断Hot-Join  = Target 后加入总线CCC       = Common Command Code，标准管理命令CRR       = Controller Role Request，Secondary Controller 请求接管总线
```

这就是 I3C 比 I2C 更像“可管理总线”的地方。规范把 IBI、Hot-Join、多 Controller、CCC、动态地址等都作为 I3C SDR 基础协议的重要组成部分。
