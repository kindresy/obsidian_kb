Sir，这一章可以按一个核心模型来学：**I3C 总线上永远只有一个当前控场者，但允许多个具备控场能力的设备；所有通信都被封装成 Frame；`7'h7E` 是 I3C 世界的“广播/入口地址”。** 规范现在把传统 **Master/Slave** 术语改成 **Controller/Target**，也就是你说的主/从设备分别对应 Controller/Target。

## 1. 总线原则：I3C 是“两线、多点、单 Active Controller”总线

I3C 仍然是两根线：

|线|作用|
|---|---|
|SDA|双向数据线|
|SCL|通常是时钟线；某些 HDR 模式下也可参与数据传输|

最重要的原则是：**同一时刻只能有一个 Active Controller 控制总线**。其他设备即使具备 Controller 能力，也必须先作为 Target 存在，等到角色交接后才能成为新的 Active Controller。规范明确说 I3C 是两线双向串行总线，面向多个 Target，由一个 Controller 在某一时刻控制；Controller 角色可以被交接给支持该能力的 Secondary Controller。

你可以把 I3C 总线理解成：

```text
一条 I3C Bus
 ├── 当前控场者：Active Controller
 ├── 普通响应者：I3C Target
 ├── 潜在控场者：Secondary Controller，平时也是 Target
 └── 可选共存者：Legacy I2C Target
```

I3C 比 I2C 多出来的关键能力包括：动态地址、In-Band Interrupt、Hot-Join、多 Controller、广播/直达 CCC、HDR 模式等。默认初始化和基础通信仍然从 **SDR Mode** 开始。

## 2. 帧结构：START → HEADER → DATA → STOP

规范对 Frame 的定义很直接：**Frame 从 START 开始，随后是 Address/Header、Data，最后 STOP**。第 4.1 节还强调：I3C Frame 至少包含 START、Header、Data、STOP；Header 后面允许做总线仲裁。

可以先记这个基本形态：

```text
START → HEADER → DATA → STOP
```

展开后更像这样：

```text
START
  ↓
HEADER = 7-bit Address + RnW + ACK/NACK
  ↓
DATA   = 若干 9-bit SDR Word 或 HDR 格式数据
  ↓
STOP
```

### START

START 和 I2C 类似：**SCL 为高时，SDA 从高跳低**。规范也保留了 `S` 作为 START 缩写。

### HEADER

Header 格式和 I2C 地址头一样：

```text
7-bit Address + RnW + ACK/NACK
```

但 I3C 的 Header 有一个关键增强：**紧跟 START 的 Header 是可仲裁的 Arbitrable Address Header**。这意味着 Controller 和某些 Target 可能同时尝试发起行为，比如：

```text
Controller 想发起普通传输
Target 想发 IBI
Secondary Controller 想请求 Controller Role
Hot-Join Target 想加入总线
```

所以 START 后的 Header 通常使用 Open-Drain 仲裁。Repeated START 后的 Header 不再仲裁，通常由 Controller 用 Push-Pull 驱动，只有 ACK/NACK 仍然是 Open-Drain。

### DATA

SDR 模式下，I3C 的 Data Word 和 I2C 一样是 9 bit，但第 9 bit 的意义不同。

Controller 写 Target 时：

```text
8-bit Data + T-bit
```

这里 T-bit 是奇校验 parity，不再是 I2C 里的 Target ACK。

Target 读返回时：

```text
8-bit Data + T-bit
```

这里 T-bit 用来表示是否继续、结束，或者让 Controller 中止读传输。也就是说，**I3C 的第 9 bit 更像流控/校验语义，不再简单等价于 I2C ACK/NACK**。

### STOP

STOP 和 I2C 类似：**SCL 为高时，SDA 从低跳高**。规范用 `P` 表示 STOP。

## 3. Controller 角色：Primary Controller vs Secondary Controller

你说的 **Main Master**，在新规范里更准确叫：

```text
Primary Controller
```

你说的 **Secondary Master**，在新规范里叫：

```text
Secondary Controller
```

再加一个运行时概念：

```text
Active Controller
```

三者关系非常关键。

|概念|含义|
|---|---|
|Primary Controller|初始配置 I3C Bus 的 Controller，是总线启动后的第一个 Active Controller|
|Secondary Controller|具备 Controller 能力，但初始作为 Target 工作|
|Active Controller|当前真正控制总线的那个 Controller，同一时刻只能有一个|

### Primary Controller

Primary Controller 的职责是初始化和配置总线，包括动态地址分配、获取 Target 能力、发送 CCC、管理 IBI/Hot-Join/Controller Role Request 等。规范说 Primary Controller 是初始化 I3C Bus、配置所有 Target 的 Controller-capable 设备，并成为配置完成后的第一个 Active Controller。

简单说：

```text
Primary Controller = 总线的初始管理者
```

它不是“永远的唯一主机”，因为 I3C 支持 Controller Role Handoff。

### Secondary Controller

Secondary Controller 是一个很特殊的角色：**它有 Controller 能力，但平时作为 Target 存在**。当它需要控制总线时，可以通过 Controller Role Request 请求成为 Active Controller。规范明确说 Secondary Controller 初始作为 Target 工作，但可以接受 Active Controller 交接来的 Controller Role；一旦接受，它就变成新的 Active Controller。

简单说：

```text
Secondary Controller = 平时是 Target，有需要时可申请变成 Controller
```

### Active Controller

Active Controller 不是一种固定设备类型，而是一个**当前状态**：

```text
谁现在控制总线，谁就是 Active Controller
```

规范还特别说明，Active Controller 不是正式的 Device Role，也不会通过 BCR 暴露。

## 4. Target 角色：不主动控总线，但可以发起“请求”

I3C Target 对应旧术语里的 Slave。规范现在把 Slave 术语废弃，统一叫 Target。Target 的基本职责是：

```text
监听总线 → 匹配地址 → 响应 Controller 的 CCC 或私有读写
```

I3C Target 至少支持 SDR Mode，并支持某种动态地址分配方式，例如 SETDASA、SETAASA 或 ENTDAA。Target 通常不产生 SCL 时钟，而是跟随 Active Controller 的时钟。

Target 又分为几类：

|角色|含义|
|---|---|
|I3C Target|普通 I3C Target，支持 SDR，且至少支持一种 HDR|
|SDR-Only Target|只支持 SDR，不支持 HDR|
|I2C Target|Legacy I2C 设备，不具备 I3C Controller/Target 能力|
|Secondary Controller|从总线视角看，未成为 Active Controller 前也表现为 Target|

Target 虽然不主动控制总线，但 I3C 给了它一些“请求机制”：

```text
IBI          → Target 请求中断
Hot-Join     → Target 请求加入总线并获取动态地址
CRR          → Secondary Controller 请求成为 Active Controller
```

这些请求都通过 START 后的 Address Arbitration 机制完成。

## 5. `7'h7E` 广播地址的含义

`7'h7E` 是 I3C 里非常核心的地址。它不是普通 Target 的动态地址，而是：

```text
I3C Broadcast Address
```

规范定义 Broadcast 是面向多个 Target 的命令，使用广播地址 `7'h7E`。并且规定：所有 I3C Target 都要匹配 `7'h7E`；Legacy I2C Target 不会匹配这个地址，因为该地址在 I2C 中是保留未使用的。

### 它的核心作用有三个

第一，`7'h7E` 用来标识 I3C 广播消息：

```text
START → 7'h7E + W + ACK → CCC → DATA → STOP
```

这里 `CCC` 就是 Common Command Code。Broadcast CCC 通过 `7'h7E` 发给所有 I3C Targets。

第二，`7'h7E` 是 I3C 模式/流程的入口信号。比如进入 HDR Mode 时，Controller 会先发 `7'h7E`，再发 Enter HDR CCC，然后进入对应 HDR 传输。规范的 Figure 7 就是用 `7'h7E` 作为 Broadcast CCC、Direct CCC 和 Enter HDR 的入口。

第三，`7'h7E` 会被所有 I3C Target ACK。这个 ACK 是 Open-Drain 的，所以多个 Target 同时 ACK 是安全的。

```text
S → 7'h7E → W → ACK
                  ↑
          所有 I3C Target 可共同拉低 ACK
```

### 注意：`7'h7E` 不能分配给普通 Target

I3C Dynamic Address 不能使用 `7'h7E`。规范的地址限制表里明确把 `7'h7E` 标为 I3C Reserved。此外，像 `7'h3E`、`7'h5E`、`7'h6E`、`7'h76`、`7'h7A`、`7'h7C`、`7'h7F` 这些和 `7'h7E` 只差一位的地址也禁止用来作为 Target 动态地址，因为它们会影响广播地址单 bit 错误检测。

## 6. 用一句话串起来

你可以这样记这一章：

```text
I3C Bus 上同一时刻只有一个 Active Controller。
Primary Controller 负责初始化总线，Secondary Controller 平时是 Target，必要时可请求接管总线。
Target 通过动态地址、广播地址或组地址被访问，也可以用 IBI/Hot-Join/CRR 发起请求。
所有通信封装成 Frame：START → HEADER → DATA → STOP。
其中 HEADER 负责地址匹配和仲裁，7'h7E 是所有 I3C Target 都要识别的广播入口地址。
```

从驱动/波形角度，看到 `S → 7'h7E → W → ACK`，你基本可以判断：**Controller 正在发起 I3C 广播/CCC/模式切换相关流程，而不是普通 I2C 访问某个从设备。**
