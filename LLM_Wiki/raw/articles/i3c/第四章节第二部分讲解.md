Sir，可以。这里我先帮你把 **HDR 高速模式**建立成一个清晰框架。严格对应 MIPI I3C Basic v1.1.1 的话，HDR 主体不是 4.2/4.3，而是在 **5.2 High Data Rate Modes**；如果你的学习材料把它叫第三章 4.2~4.3，我们就按“技术概述里的 HDR 高速模式”来学。

# 1. HDR 的核心定位：SDR 是管理面，HDR 是高速数据面

I3C 的总线永远先在 **SDR Mode** 初始化和配置，不会一上电就进入 HDR。动态地址分配、Hot-Join、IBI、Controller Role Handoff、很多总线配置类 CCC，仍然属于 SDR 管理面。HDR 是在 SDR 基础上临时进入的高速传输模式。

可以这样理解：

```text
SDR:
  总线初始化
  Dynamic Address Assignment
  CCC 基础管理
  IBI / Hot-Join / CRR
  普通 Private Read / Write

HDR:
  高速数据传输
  更高吞吐
  特定格式的 Command / Header / Data / CRC
  通过 HDR Exit Pattern 回到 SDR
```

一句话：**SDR 是控制平面，HDR 是高速数据平面。**

---

# 2. I3C Basic 里的 HDR 模式有哪些？

I3C Basic v1.1.1 包含两个 HDR 模式：

| HDR 模式  | CCC 入口  | 核心特点                            |
| ------- | ------- | ------------------------------- |
| HDR-DDR | ENTHDR0 | 双沿传输，Command/Data Word 模型       |
| HDR-BT  | ENTHDR3 | Bulk Transport，块传输，适合大数据和多 Lane |

完整 I3C 规范里还有 HDR-TSP / HDR-TSL，但 **I3C Basic 不包含这两个 Ternary 模式**。

你可以先把它们记成：

```text
HDR-DDR = I3C 两线高速字传输
HDR-BT  = I3C 高速块传输 / Bulk Transport
```

---

# 3. HDR 的通用进入流程

所有 HDR 都不是凭空进入的，必须先通过 SDR 发送 Enter HDR CCC。

典型入口：

```text
S / Sr
7'h7E + W + ACK
ENTHDRx CCC + T
进入 HDR Mode
```

比如进入 HDR-DDR：

```text
S
7'h7E + W + ACK
ENTHDR0
T
HDR-DDR Command / Data / CRC ...
HDR Exit Pattern
P
```

进入 HDR-BT：

```text
S
7'h7E + W + ACK
ENTHDR3
T
HDR-BT Header / Data Blocks / CRC ...
HDR Exit Pattern
P
```

这里的关键点是：

```text
7'h7E + W 仍然是 SDR 格式
ENTHDRx 仍然是 SDR CCC
ENTHDRx 后面才切到 HDR
```

所以抓波形时，你会看到 **SDR 前导 + HDR 主体**。

---

# 4. HDR 的通用结束：HDR Restart vs HDR Exit

进入 HDR 后，不再靠普通 SDR 的 Repeated START / STOP 来分割所有内容，而是靠 HDR 专用 Pattern。

## 4.1 HDR Restart Pattern

作用类似 SDR 里的 Repeated START：

```text
当前仍保持 HDR Mode
结束当前 HDR transfer
马上开始下一个 HDR transfer
```

形式上：

```text
HDR Transfer 1
HDR Restart Pattern
HDR Transfer 2
HDR Restart Pattern
HDR Transfer 3
...
```

注意：**HDR Restart 不退出 HDR**。

---

## 4.2 HDR Exit Pattern

作用是退出 HDR，回到 SDR。

```text
HDR Transfer
HDR Exit Pattern
STOP
Bus Free
回到 SDR
```

规范强调所有 I3C Target 都必须能检测 HDR Exit Pattern，即使这个 Target 本身不支持当前 HDR 模式。原因是：如果 Controller 进入一个某 Target 不支持的 HDR 模式，这个 Target 至少要能“装睡”等退出，不至于误解后续总线波形。

这点对 RTL/验证很重要：

```text
Target 可以不支持 HDR-DDR / HDR-BT
但必须具备 HDR Exit Pattern Detector
```

---

# 5. HDR-DDR：双沿高速 Word 协议

HDR-DDR 的本质是：

```text
SCL 仍由 Controller 驱动
SDA 在每个 SCL 边沿之后变化
接收方在后续边沿采样
```

所以它比 SDR 更快，因为 SDR 主要是一拍一个 bit，而 HDR-DDR 利用双沿传输。

## 5.1 HDR-DDR 传输结构

典型 HDR-DDR Frame：

```text
SDR:
  S / Sr
  7'h7E + W + ACK
  ENTHDR0 + T

HDR-DDR:
  Command Word
  Data Word(s)
  CRC Word
  HDR Restart Pattern 或 HDR Exit Pattern

若 Exit:
  P
```

规范里的典型例子就是：ENTHDR0 进入 HDR-DDR 后，Controller 发 Command Word，然后是一个或多个 Data Word，最后 CRC Word；之后可以 HDR Restart 继续下一笔，也可以 HDR Exit + STOP 结束。

---

## 5.2 HDR-DDR Word 格式

HDR-DDR 的 Command Word / Data Word 是 **20 bits**：

```text
2-bit Preamble
16-bit Payload
2-bit Parity
```

也就是：

```text
[Preamble] [Payload 15:0] [PA1 PA0]
```

有效 payload 是 16 bit，所以如果 SCL 是 12.5 MHz，raw rate 是 25 Mbps，但考虑 20 bit 里只有 16 bit payload，实际数据效率大约是：

```text
16 / 20 = 80%
```

也就是有效 payload rate 约 20 Mbps。

---

## 5.3 HDR-DDR Read / Write

### Write：Controller → Target

```text
Controller:
  Command Word
  Data Word(s)
  CRC Word

Target:
  接收并校验
```

### Read：Target → Controller

```text
Controller:
  Command Word

Target:
  Data Word(s)
  CRC Word

Controller:
  HDR Restart 或 HDR Exit
```

注意：**Command Word 永远由 Controller 发**。Data Word 根据方向由 Controller 或 Target 发。SCL 在 HDR-DDR 中仍然由 Controller 驱动。

---

# 6. HDR-BT：Bulk Transport，块传输模式

HDR-BT 是 I3C Basic 里更“数据搬运型”的高速模式，尤其适合：

```text
大量数据
宽内部总线
SRAM / FIFO / AXI-like 数据搬运
多 Lane 传输
```

它的思路不是一个个 16-bit Word 地传，而是用更大的块结构。

---

## 6.1 HDR-BT 总结构

典型 HDR-BT Frame：

```text
SDR:
  S / Sr
  7'h7E + W + ACK
  ENTHDR3 + T

HDR-BT:
  Header Block
  Data Block(s)
  CRC Block
  HDR Restart Pattern 或 HDR Exit Pattern

若 Exit:
  P
```

规范描述 HDR-BT transfer 从 Header Block 开始，然后一个或多个 Data Block，最后 CRC Block；多个 transfer 可通过 HDR Restart 串起来，最后 HDR Exit + STOP 结束。

---

## 6.2 HDR-BT 的结构元素

HDR-BT 主要分三层：

```text
Header Block
N * Data Blocks
CRC Block
```

### Header Block

Header Block 是 7 字节：

```text
Byte0: Address
Byte1: Cmd0
Byte2: Cmd1
Byte3: Cmd2
Byte4: Cmd3
Byte5: Control
Byte6: Transition
```

其中 Address 字节可以是：

```text
Target Dynamic Address + R/W
Group Address + W
7'h7E 广播地址，用于 HDR-BT 中的 CCC framing
```

Cmd0~Cmd3 通常可作为 command / index / offset / length，具体语义由 Controller 和 Target 的功能协议约定。

### Data Block

HDR-BT 的数据块模型是：

```text
32 bytes data + 1 byte transition/control
```

最后一个 Data Block 可以不是满 32 字节，也就是所谓 ragged last block。

### CRC Block

HDR-BT 结尾用 CRC 校验，可以是 CRC-16 或 CRC-32，由 Header 里的控制信息指示。

---

## 6.3 HDR-BT 支持 Single / Dual / Quad Lane

HDR-BT 可以在：

```text
Single Lane
Dual Lane
Quad Lane
```

上工作。

这也是它比 HDR-DDR 更适合高吞吐数据搬运的原因。Single Lane 下它也可以用，但真正优势在多 Lane。规范也说 HDR-BT 在 Single Lane 下仍然可用，只是搬同样数据需要更多 clock。

---

# 7. HDR 模式下 CCC 怎么办？

HDR 里也可以发送某些 CCC，但不是所有 CCC 都允许。

核心原则：

```text
会改变总线根本配置的 CCC，通常只能在 SDR 做
```

比如这些不允许在 HDR 中做：

```text
RSTDAA
ENTDAA
ENTHDR0~ENTHDR7
SETAASA
SETDASA
SETNEWDA
GETACCCR
```

原因很直接：

```text
动态地址分配只能在 SDR
进入 HDR 只能从 SDR 进入
Controller Handoff 只支持 SDR
```

而某些查询类或配置类 CCC 可以在 HDR 中被允许，例如 GETSTATUS、GETMXDS、GETCAPS 等，但 Target 对“某个 CCC 在某个 HDR 模式下”的支持通常是可选的。Controller 必须知道哪些 Target 支持当前 HDR Mode 以及当前 HDR CCC。

---

# 8. HDR 和 IBI / Hot-Join 的关系

这是很容易混的点。

IBI、Hot-Join、Controller Role Request 的典型机制依赖 SDR 的 START、Address Arbitration、Target 拉 SDA 请求 START 等行为。进入 HDR 后，总线处在 HDR 专用 framing 里，所以这些事件一般不是在 HDR 内直接发生。

你可以先这样记：

```text
IBI / Hot-Join / CRR 是 SDR 管理事件
HDR 是高速传输窗口
Target 要发 IBI，通常要等 Controller 退出 HDR 回 SDR
```

所以 ENEC/DISEC 即使某些情况下可在 HDR 里使用，也“不一定有实际意义”，因为被控制的事件类型需要回 SDR 后才能自然发生。

---

# 9. SDR vs HDR-DDR vs HDR-BT 对比

|项目|SDR|HDR-DDR|HDR-BT|
|---|---|---|---|
|入口|默认模式|ENTHDR0|ENTHDR3|
|用途|管理 + 普通读写|高速 Word 传输|高速块传输|
|初始化/DAA|支持|不支持|不支持|
|IBI/Hot-Join|支持|不作为主要事件模式|不作为主要事件模式|
|数据结构|8-bit + T-bit|20-bit Word|Header/Data/CRC Blocks|
|校验|SDR parity / T-bit|parity + CRC-5|CRC-16 / CRC-32|
|多 Lane|SDR Multi-Lane 不属于 I3C Basic|I3C Basic 不含 HDR-DDR Multi-Lane|支持 HDR-BT Multi-Lane|
|退出方式|STOP / Sr|HDR Exit / Restart|HDR Exit / Restart|

---

# 10. 从驱动/验证角度看 HDR 主流程

一个比较真实的 Controller 侧流程是：

```text
1. SDR 初始化总线
   RSTDAA / ENTDAA / SETDASA / SETAASA

2. 枚举 Target 能力
   GETBCR / GETDCR / GETCAPS / GETMXDS

3. 判断 Target 是否支持 HDR-DDR 或 HDR-BT

4. 配置必要参数
   max speed
   max read/write length
   HDR-BT lane coding
   CRC 能力

5. SDR 下发送 ENTHDRx
   ENTHDR0 = HDR-DDR
   ENTHDR3 = HDR-BT

6. 进入 HDR
   发送 HDR Command 或 Header

7. 传输 Data
   HDR-DDR: Data Word(s)
   HDR-BT : Data Block(s)

8. 校验
   HDR-DDR: CRC Word
   HDR-BT : CRC Block

9. 选择继续或退出
   HDR Restart Pattern → 继续下一笔 HDR transfer
   HDR Exit Pattern + P → 回到 SDR
```

---

# 11. 最容易踩的坑

第一，不要把 HDR Restart 当成 SDR Repeated START。它们语义相似，都是“不中断总线继续下一笔”，但编码完全不同。

第二，不要在 HDR 中做 DAA。动态地址分配、地址变更、Controller Role Handoff 这类根本配置动作应在 SDR 完成。

第三，不要以为 Target 不支持 HDR 就会崩。正确的 Target 即使不支持当前 HDR，也应该等待 HDR Exit Pattern。

第四，HDR-DDR 和 HDR-BT 不是同一种“更快 SDR”。HDR-DDR 是 Word 模型，HDR-BT 是 Block/Bulk 模型，驱动里的 descriptor、FIFO、CRC、DMA 组织方式都会不同。

第五，HDR-BT 的优势不只是速度，而是更适合块数据搬运，尤其当内部总线是 AXI、SRAM、FIFO 或者需要 32B/64B 粒度处理时。

---

# 12. 这一节你先抓住的主线

```text
I3C HDR 高速模式主线：

SDR 初始化总线
  ↓
通过 7'h7E + ENTHDRx 进入 HDR
  ↓
HDR-DDR 用 Command/Data/CRC Word 传输
HDR-BT 用 Header/Data Block/CRC Block 传输
  ↓
用 HDR Restart 继续下一笔 HDR
或用 HDR Exit + STOP 回到 SDR
```

下一步最值得细攻的是 **HDR-DDR 的 Command Word / Data Word / CRC Word / ACK-NACK / Read-Write flow**，因为它比 HDR-BT 更接近你前面学的 SDR，也更适合作为 HDR 的第一突破口。
