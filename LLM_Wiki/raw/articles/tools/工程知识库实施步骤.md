# Engineering Knowledge Compiler — 实施步骤

基于 [[工程知识库搭建思路]] 提取的阶段性实施计划。

---

## 总体架构（四层）

```
L4: Query / Teaching / Dashboard       ← 查询 + 教学图谱 + 可视化
                ↑
L3: Binding Graph                        ← 概念 ↔ 代码符号 ↔ Flow ↔ 文档声明
                ↑
L2: Code Graph                           ← 函数/文件/模块/调用/包含/操作
                ↑
L1: LLM Wiki                             ← raw/ → wiki/ → index/log/compile/promote
```

## Phase 2：文档概念 ↔ 代码实现绑定（待实施）

**目标**：把 RAW/Wiki 里的概念绑定到代码符号。

### 四级绑定

#### Level 1：确定性绑定
直接匹配函数名、结构体名、宏名、寄存器名、文件路径。
```
Wiki 中出现 i2c_dw_handle_tx_abort()
  → 直接绑定 CodeSymbol
```
**Confidence**: 高，自动通过。

#### Level 2：别名/术语绑定
驱动开发中概念和代码名往往不一致：
```
文档概念        代码实现
TX abort       tx_abrt
NACK           no_ack / ack_fail
BAR0           pci_resource_start(pdev, 0)
```
需要维护别名表：
```yaml
# concept frontmatter
aliases:
  - tx abort
  - tx_abrt
  - TX_ABRT
code_patterns:
  - "*tx*abrt*"
  - "*abort*source*"
```

#### Level 3：语义候选绑定
用 embedding + 代码摘要搜索 top-k 候选，LLM 判断关系类型，生成 evidence。
**所有候选进入 pending review，不自动 promote。**

#### Level 4：人工确认绑定
非确定性绑定写入 `pending/code-bindings.md`，人工审查后 promote。

### 实施步骤

- [ ] 创建绑定数据库 `bindings.sqlite`
- [ ] 实现 Level 1：精确符号匹配
- [ ] 实现别名表机制（concept frontmatter 或独立文件）

- [ ] 在概念页中增加 `## Implementations` 区块

## Phase 4：Query + 教学图谱

**目标**：统一查询、可视化教学、代码带读。

### Hybrid Query

```
用户问题
  → 识别概念实体
  → 查 Wiki 页面
  → 查 Binding Graph（概念→代码符号）
  → 查 CodeGraph（callers/callees/flow）
  → LLM 综合回答
```

### 优先支持的查询

1. 某个文档概念在代码里由谁实现？
2. 某个寄存器在哪些路径被读写？
3. 某个错误码从硬件一路怎么传到上层？
4. 某个 probe/remove/suspend/resume flow 怎么走？
5. 修改某个函数会影响哪些 flow？
6. 文档说法和代码实现是否矛盾？
7. 新人应该按什么顺序阅读这个 driver？

## 总体时间线

```
Phase 1: 代码仓库 → Wiki     ✅ 已完成（可迭代优化）
Phase 2: 概念 ↔ 代码绑定      ⬜ 下一阶段
Phase 3: Flow 页面生成         ⬜ 与 Phase 2 并行
Phase 4: Query + 教学图谱     ⬜ 最后阶段
```

## 关键原则

- **先准，再全**：非确定性绑定必须进 pending，禁止自动 promote
- **先可追溯，再可视化**：每条 binding 必须带 evidence
- **先绑定，再教学**：没有可靠绑定之前，教学图谱没有意义
- **代码结果是 snapshot**：必须带 repo_name + commit_sha + index_time
- **lint 必须检查 stale**：代码变了但 Wiki 没更新，要能报出来
